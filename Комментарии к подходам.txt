Сравнение подходов к сериализации данных
1. ООП стиль: Соблюдение инкапсуляции (Safe)
   - Работа с объектом только через его публичный интерфейс (свойства @property).
   - Если внутренняя реализация класса изменится (например, _name станет _full_name), код не сломается, так как публичный метод name() останется прежним. Гарантируется валидность объекта (через __init__).
   - Требует наличия публичных геттеров для всех полей.

2. ООП стиль: Нарушение инкапсуляции (Unsafe)
   - Игнорируется методы, читение_private поля напрямую. Создание объекта через __new__, минуя конструктор.
   - Работает с любыми классами, даже закрытыми (нет лишних вызовов).
   - При любом изменении имен переменных внутри класса сериализатор сломается. Опасность создать битый объект, не прошедший проверки конструктора.

3. Функциональный стиль: Соблюдение инкапсуляции
   - Функции serialize/deserialize, принимающие объект, используют публичные геттеры.
   - Чистые данные отделены от логики сохранения.
   - Зависимость от публичного API.

4. Функциональный стиль: Нарушение инкапсуляции (через __dict__)
   - Использование интроспекции Python (obj.__dict__) для мгновенного дампа всего состояния объекта.
   - Вместо перечисления полей (_name, _friends), идет копирование всей памяти объекта. Это делает код универсальным, но очень зависимым от реализации CPython.
